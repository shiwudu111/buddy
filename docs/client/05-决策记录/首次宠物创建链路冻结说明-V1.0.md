# 首次宠物创建链路冻结说明-V1.0

## 文档信息
- **版本**：V1.0
- **创建时间**：2026-04-14
- **适用仓库**：`buddy-client`
- **适用范围**：学生端“无宠物账号首次进入”创建链路
- **状态**：已生效，作为 MVP 阶段首次宠物创建链路的执行基线

## 一、文档目的
本文用于冻结学生端“无宠物账号首次进入”的正式产品口径，避免前端原型、后端契约、测试验收三方继续漂移。

本文基于当前项目讨论与《学伴精灵MVP功能范围确认》整理；该文件明确说明自己是“参考副本，非权威源”。如与更高优先级的冻结决策、API 基线或数据库文档冲突，以更高优先级文档为准。

当前 MVP 核心宠物能力冻结在“宠物创建、基础信息、资源系统、基础成长逻辑”，执行关注点强调“主链路优先”和“接口对齐”。

## 二、本次冻结结论

### 2.1 正式冻结能力
MVP 阶段，首次宠物创建链路正式冻结为：

`无宠物学生账号首次进入主界面 -> 触发首次创建入口 -> 输入宠物名称 -> 调用创建宠物接口 -> 创建成功 -> 进入成长主链路`

这里的正式业务结果只有一个：

**账号成功创建一只宠物，并进入后续成长主链路。**

### 2.2 当前不冻结的内容
以下内容不属于本期正式业务结果，不得作为正式承诺进入开发、联调或验收口径：

- 蛋型
- 稀有度
- 物种
- `hatchSeed` / 随机种子
- 孵化结果
- 抽卡式动画结果
- “选择某个蛋会决定最终宠物”的业务语义

换句话说，当前 MVP 冻结的是“创建宠物”，不是“正式选蛋孵化系统”。

这与现有 MVP 范围文档中“宠物创建”为核心功能、“用户可以创建和查看宠物”为验收标准的定义一致。

## 三、产品口径说明

### 3.1 当前产品定义
当前阶段，学生端首次进入时可以存在轻量引导、命名输入、欢迎演出，但这些内容仅能服务于“首次创建宠物”体验本身，不能向用户表达为一套已冻结的正式孵化业务。

### 3.2 原型与正式业务的边界
若客户端已存在“森林蛋 / 星空蛋 / 海洋蛋”或类似演出内容，现阶段统一定义为：

**前端演出原型，不代表后端真实业务结果。**

因此：

- 演出可以存在，但不能宣称“选择结果会决定最终宠物”
- 演出不得反向定义正式数据模型
- 演出不得进入正式验收标准
- 演出默认关闭，或挂在 `feature flag` 下受控启用

## 四、前端实现边界

### 4.1 本期允许实现
前端本期可实现以下内容：

- 学生账号首次进入时的无宠物判定
- 首次创建入口页或遮罩页
- 宠物名称输入
- 创建请求发起
- 创建成功后的轻量反馈演出
- 创建失败后的提示与回退
- 创建完成后进入主界面正常宠物成长链路

### 4.2 本期不建议继续实现
以下内容不应继续按正式功能推进：

- 选蛋决定结果
- 本地 `rarity` 计算
- 本地 `species` 决策
- 本地 `hatchSeed` 持久化
- 多步抽卡式 onboarding
- 将前端本地孵化模型写成正式业务模型

### 4.3 模块边界
首次创建流程不再建议继续堆叠进 `MainController`。

建议独立为 `PetOnboarding`、`FirstPetFlow` 或同级模块。

原因很明确：

“首次进入判定、无宠物检测、引导、命名、创建请求、失败回退、成功切主链路”本质上已经是一个独立流程，不应继续污染主界面控制器职责。

## 五、后端契约边界

### 5.1 当前正式契约
MVP 当前正式契约仍按已有创建宠物能力执行，即：

**创建宠物接口只保证完成宠物实体创建。**

若当前后端能力仍为 `createPet(name)`，则该契约在本期继续有效；前端不得擅自将 `eggType`、`species`、`rarity`、`hatchSeed` 等字段包装为正式业务输入。

### 5.2 当前后端不承诺的内容
后端本期不承诺：

- `eggType`
- `species`
- `rarity`
- `hatchSeed`
- 孵化结果可复现
- 蛋型影响最终结果

### 5.3 后端文档建议补充
建议在 API 基线或冻结决策中增加一条说明：

**当前 MVP 的宠物创建接口仅保证创建宠物实体；不承诺 `eggType`、`species`、`rarity`、`hatchSeed` 等扩展业务字段。**

这样可以避免联调阶段出现“前端演出像正式功能、后端实际并未支持”的伪完成状态。

## 六、测试验收口径

### 6.1 本期正式验收范围
测试本期只验以下链路：

- 无宠物学生账号首次进入时，能够稳定触发首次创建入口
- 输入合法名称后，可以成功创建宠物
- 创建成功后，可以进入主链路并在主界面看到宠物
- 创建失败时，有明确提示、可回退、可重试
- 已有宠物账号再次进入时，不会重复误触发首次创建流程

### 6.2 本期明确不验
本期不进入正式验收的内容包括：

- 蛋型是否影响最终结果
- 演出与最终结果是否一致
- 稀有度是否稳定
- 物种是否由某个蛋型决定
- 结果是否可复现、可追踪、可回放

这些内容只有在后续被升级为正式功能，并补齐正式契约后，才进入测试验收。

## 七、数据与可测性要求
为了保证链路可验证、可定位、可回归，本期至少需要准备以下测试条件：

### 7.1 测试账号
- 无宠物学生账号
- 已有宠物学生账号

### 7.2 可观测状态
客户端或联调日志中，至少应能确认以下信息：

- 是否命中“首次进入且无宠物”判定
- 是否展示 onboarding / 首次创建入口
- 创建请求参数
- 创建结果返回内容
- 创建失败原因
- 创建成功后主界面宠物状态是否已刷新

没有这些可观测信息，就只能“看 UI 猜真相”，无法证明链路真正稳定。

## 八、Feature Flag 约束
若客户端保留现有“选蛋 / 孵化演出”原型，则必须满足以下约束：

- 默认关闭，或仅在内部演示环境开启
- 不作为正式版本默认路径
- 不写入正式验收 case
- 不写入正式业务结论
- 文档需明确标注“仅演示，不代表后端真实结果”

## 九、后续升级条件
若后续决定将“选蛋孵化”升级为正式功能，则必须先完成以下冻结，再进入开发：

### 9.1 业务权威
最终宠物结果必须只有一个权威来源。

建议采用：**后端权威**。

### 9.2 正式数据模型
至少明确：

- `eggType`
- `species`
- `rarity`
- `hatchSeed`
- 生成规则
- 持久化方式

### 9.3 接口契约
至少明确：

- 入参是否包含 `eggType` / `seed`
- 哪些字段是客户端可传
- 哪些字段由后端生成
- 返回结果是否为最终真相

### 9.4 验收标准
至少明确：

- 测试到底验“创建成功”
- 还是验“蛋型影响结果”
- 还是验“结果可复现、可追踪、可回放”

在这些内容未冻结前，“选蛋孵化”不得升格为正式开发项。

## 十、最终会议结论
本次会议统一结论如下：

1. 当前问题不是继续增加宠物蛋 UI，而是先收口“首次创建宠物”正式定义。
2. MVP 本期正式冻结能力为：无宠物学生首次进入 -> 创建宠物 -> 进入成长主链路。
3. 蛋型、稀有度、物种、孵化结果、抽卡动画等内容，当前均不属于正式业务结果。
4. 客户端已有选蛋/孵化内容暂定为原型演出，默认关闭或挂 `feature flag`，不进入正式验收。
5. 首次创建链路应独立为 onboarding 模块，不再继续堆进 `MainController`。
6. 若后续要升级为正式孵化功能，必须先冻结业务权威、接口契约、数据模型与测试标准再开发。

## 十一、与现有 MVP 文档的一致性说明
该冻结说明与现有 MVP 范围文档保持一致：

当前核心目标是保证主链路跑通，宠物能力聚焦于“宠物创建”和基础成长，不扩展未冻结的复杂宠物生成语义；同时强调 server / client / 文档三方遵守 API 基线与范围控制。
