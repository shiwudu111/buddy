# 登录入口重构变更决议-V1.0

## 文档信息
- **版本**：V1.0
- **创建时间**：2026-04-10
- **适用仓库**：`buddy-client`
- **状态**：已生效，作为登录入口重构执行基线

## 一、变更背景
当前登录入口已经完成了基础工程化拆分，学生端与家长端主链路也已联调跑通；但现有登录页仍然建立在“单页表单模型”上，即同屏承载：
- 身份判断
- 登录/注册行为判断
- 账号密码输入

该模型已不再适合新的登录入口方案。新的视觉与产品方案本质上不是“登录页美化”，而是“登录入口流程重构”，需要将登录入口改造成：

`restore -> brandEntry -> roleSelect -> authForm`

本次变更的目标，是在不打乱现有主链路与后端接口基线的前提下，完成从“单页表单模型”到“多步骤入口模型”的切换。

## 二、变更结论

### 2.1 正式结论
本次工作正式定义为：

**登录入口流程重构**

而不是：

**登录页视觉优化**

### 2.2 决策判断
1. 该变更属于产品入口层重排，必要性高，应立即纳入正式项目变更。
2. 该变更不改变 MVP 核心业务目标，不属于范围膨胀。
3. 该变更会直接替代当前旧登录页的入口形态与部分 UI 固化假设。
4. 第一阶段不要求后端新增接口，现有认证接口足以支撑入口重构。

## 三、变更范围

### 3.1 本次变更覆盖
- `Login.scene` 的入口流程模型
- 登录入口状态机
- 身份选择流程
- 登录/注册表单入口组织方式
- 登录入口相关联调口径与验收口径

### 3.2 本次变更不覆盖
- `Main.scene` 内部逻辑
- 学生端与家长端主界面结构
- 多 Scene 登录入口拆分
- 多账号管理与多账号切换
- 新增后端业务接口

## 五、正式冻结的新入口模型

### 5.1 第一阶段状态机
第一阶段正式冻结为：

`restore -> brandEntry -> roleSelect -> authForm(role, mode)`

### 5.2 第一阶段页面边界
- 保持 `Login.scene` 作为唯一登录入口场景
- 保持 `Login -> Main` 场景路由边界
- 不在第一阶段拆分多个 Cocos Scene

### 5.3 第一阶段默认规则
1. `roleSelect -> authForm` 默认进入 `mode=login`
2. 注册通过 `authForm` 内显式切换完成
3. 登录成功后的真实身份以后端返回的 `user.role` 为准
4. `restore` 无会话或恢复失败，统一回到 `brandEntry`
5. `restoreSuccess` 第一阶段不再自动跳转 `Main`，而是回到 `brandEntry` 并带上可续用账号，由用户点击“进入”继续
6. `authForm` 点击返回时，统一回到 `brandEntry`，不再回退到 `roleSelect`

## 六、客户端与后端对齐口径

### 6.1 已确认无需新增接口
第一阶段后端原则上不新增接口，现有接口足以支撑本次变更：
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `GET /api/v1/auth/me`

### 6.2 已对齐的认证口径
1. 注册接口正式支持 `role=CHILD | PARENT`
2. 登录接口成功后返回统一 `user` DTO，且包含 `role`
3. 登录页里先选“学生/家长”只用于前端流程分流，不作为登录接口参数
4. 登录成功后，身份、跳转与会话写入一律以后端返回的真实 `user.role` 为准

### 6.3 已对齐的恢复口径
以下情况，客户端统一视为“无可恢复登录态”，直接回到 `brandEntry`：
- `/auth/me` 返回 `401`
- token 无效
- 服务端明确判定当前会话不可恢复

说明：
- 该语义在客户端实现上应映射为 `restoreNoSession`
- 不再复用旧单页表单中的 `idle`

### 6.4 已对齐的注册失败口径
后端继续保持稳定错误语义，至少覆盖：
- 用户名已存在
- 参数缺失或参数不合法

客户端据此在表单内做就近错误提示，不依赖模糊自由文案猜测错误类型。

## 七、实施顺序决议
本次变更的实施顺序正式冻结为：

1. 先冻结状态机与实施方案
2. 再拆分流程状态层与认证动作层
3. 再改 `Login.scene` 内部多容器结构
4. 再进行联调、回归与验收
5. 最后统一更新基线文档与执行单

补充要求：
- 在新流程跑通前，不继续给旧三按钮登录页做增量优化
- 在新流程跑通前，不直接覆盖旧冻结文档正文

## 八、代码层落地原则

### 8.1 保留的模块边界
- `LoginAuthCoordinator` 继续只负责：
  - 恢复登录态
  - 登录
  - 注册

### 8.2 需要新增的层
- `LoginFlowState`
- `LoginFlowCoordinator` 或同级流程协调层

### 8.3 需要改造的层
- `LoginController`
  - 从单页表单控制器转为入口流程编排控制器
- `LoginViewOrchestrator`
  - 从单表单面板编排器转为多容器编排器
- `LoginSceneStructure`
  - 从单表单骨架转为入口层 / 身份层 / 表单层骨架

## 九、验收口径决议
新登录入口的第一阶段验收至少覆盖：
1. 有效学生登录态启动后回到 `brandEntry`，并可通过默认账号直接续入 `Main`
2. 有效家长登录态启动后回到 `brandEntry`，并可通过默认账号直接续入 `Main`
3. 无登录态时先进入 `brandEntry`
4. 选择学生后默认进入学生登录表单
5. 选择家长后默认进入家长登录表单
6. 登录成功后按真实 `role` 进入正确端
7. 注册成功后按注册 `role` 进入正确端
8. `/auth/me` 返回 `401` 或 token 无效时稳定回到 `brandEntry`
9. 登录/注册失败时停留在当前表单，并显示就近错误提示
10. 认证表单点击返回时，稳定回到账号入口页
