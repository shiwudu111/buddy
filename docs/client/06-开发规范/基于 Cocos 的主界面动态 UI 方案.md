基于 Cocos 的主界面动态 UI 方案
一、方案定位

本项目的主界面不采用传统“整页静态场景预摆”的方式，而采用 Cocos 运行时动态构建 UI 的方案。

核心原则是：

以 单主区 + 左右可收缩卡片 为基础结构
除宠物本体外，大部分 UI 通过程序运行时生成
不依赖完整页面预摆，也不依赖大量 UI 切图
通过统一的尺寸规则、组件工厂和适配策略，支持不同横屏设备

这意味着，本项目的 UI 文档重点不应是“页面切图说明”，而应是：

运行时构建规范 + 适配规则 + 组件工厂 + 资源策略

二、文档应如何定义

这份文档建议分成 4 个层级：

1. 项目摘要
给制作人、决策层看，说明这不是按机型逐台适配，而是统一规则驱动的动态 UI 方案。

2. 执行方案
给 PM、统筹看，明确开发顺序、职责分工、验收节点，以及哪些阶段不依赖美术资源。

3. 设计与资源规范
给设计、美术看，明确哪些 UI 由程序生成，哪些内容必须保留为美术资源。

4. 程序实现规范
给客户端程序看，明确节点结构、适配规则、令牌体系、组件工厂、状态管理和资源加载方式。

三、Cocos 下的适配方式

在 Cocos 中，不直接使用 Web 的 rem / vw / clamp，但应保留同样的设计思想，转换为以下三类逻辑单位：

1. 视口比例单位
按当前可视区域宽高计算，用于主区宽度、卡片展开宽度、顶部栏高度等。

2. 根单位（Root Unit）
作为统一尺寸基准，用于字号、间距、圆角、按钮高度、面板内边距等。

3. 上下限约束（Clamp）
用于限制卡片宽度、顶部栏高度、底部按钮尺寸、主舞台宽度等，避免大屏失控或小屏拥挤。

一句话概括就是：

Cocos 不用 CSS 单位，但采用等价的“比例单位 + 根单位 + 上下限约束”完成适配。

四、页面结构定义

页面应按“运行时宿主结构”描述，而不是按静态摆放图描述。

推荐结构如下：

CanvasRoot
SafeAreaRoot
BackgroundLayer
TopBarLayer
MainViewport
MainStage
LeftCardHost
RightCardHost
BottomDockLayer
OverlayLayer
PopupLayer

其中：

MainStage 是唯一主区
LeftCardHost / RightCardHost 只是左右卡片的挂点
卡片、弹窗、输入框、日志等内容都在运行时创建
场景中只保留最小壳层，不预先摆满页面节点
五、资源与程序边界
1. 建议程序动态生成的部分

以下 UI 建议默认由程序生成：

面板、卡片、弹窗底板
顶栏、底部操作区
按钮、输入框、标签、Tab
滚动区、列表项、状态条、进度条
文字、数值、日志内容

实现方式可以是：

Graphics 绘制
通用九宫格底板
少量复用底图 + 统一样式令牌
2. 建议保留为美术资源的部分

建议保留资源投入的内容主要是：

宠物主体
宠物动作 / 表情 / 配件
主题化背景或氛围资源
后期如需品牌化，可追加皮肤资源
3. 第一阶段建议

第一阶段可先完全不依赖大部分 UI 美术资源，只保留宠物占位内容即可，优先完成：

结构适配
动态布局
组件生成
页面状态流转

这样可以显著减少前期等待美术导致的阻塞。

六、设计基准与视口分类
1. 设计基准

统一以 1920 × 1080 横屏 作为设计基准。

但需要明确：

设计基准只用于规范参考，不代表运行时按固定像素还原。

2. 适配分类

不按设备型号适配，而按视口能力分类：

宽度等级：S / M / L / XL
高度等级：紧凑 / 常规 / 高
比例等级：超宽 / 常规 / Pad 型

程序运行时根据当前视口自动判断等级，再生成对应布局参数。

七、最大宽度限制必须保留

大屏或桌面窗口下，如果没有最大宽度限制，界面会被过度拉伸。

因此建议定义三类上限：

页面主容器最大宽度
主舞台最大宽度
底部操作区最大排布宽度

处理原则不是“整页缩放”，而是：

达到上限后内容居中，两侧留白，核心区不继续横向拉大。

八、推荐实现结构

程序工程建议拆分为以下模块：

core：视口、适配、令牌、主题
factory：按钮、面板、文本、输入框、滚动区等组件工厂
layout：顶栏、主区、左右卡、底部坞布局逻辑
pages：页面入口、页面状态、页面数据结构
widgets：日志卡、状态卡、宠物舞台、底部操作条等业务组件

这样做的优点是：

布局逻辑独立
组件生成独立
页面逻辑独立
后期换皮肤或扩页面，不需要推翻整套结构
九、左右卡片的定义

左右卡片不应视为静态装饰，而应定义为状态组件。

建议统一支持三种状态：

expanded
peek
collapsed

推荐默认策略：

紧凑横屏：左侧 peek，右侧 collapsed
中等横屏：左侧 expanded，右侧 peek
Pad 横屏：两侧可同时 expanded

同时要明确规则：

手机横屏同一时间只建议一侧 fully expanded
Pad 横屏可允许双侧展开
卡片展开时不应让主舞台中心明显跳位
卡片以覆盖边缘为主，不改变主区核心锚点
十、运行时创建不等于频繁重建

动态 UI 的正确理解是：

首次进入页面时创建
状态切换时复用
高频列表项用 NodePool
不使用时隐藏或回收
避免每次打开都重新 new 全套节点

建议节点池优先用于：

日志列表项
按钮项
提示条
通用弹窗
小标签项
十一、宠物层单独处理

宠物是这个页面中最值得保留美术扩展能力的部分，应与普通 UI 分层。

建议明确：

宠物节点独立于 UI 层
UI 布局不依赖宠物图片尺寸写死
主舞台只给宠物预留安全区
第一阶段可以先用占位图
动作、表情、换装等能力先定义接口，后续再接正式资源
十二、推荐开发顺序

本项目不适合“先全量高保真、再切图、再开发”的传统流程。

建议顺序如下：

第一步：先定运行时架构
确定页面结构、适配规则、令牌体系、程序生成范围和资源边界。

第二步：做低保真运行时原型
优先验证动态生成、左右卡切换、底部布局和多档视口适配是否成立。

第三步：确定组件令牌
统一字号、间距、圆角、卡片宽度、按钮尺寸等规则。

第四步：再做视觉精修
在结构稳定后决定是否增加拟物、奶油风、皮肤化表现。

第五步：最后补必要资源
重点补宠物资源、背景装饰、主题皮肤，而不是一开始就做全量切图。

十三、最终结论

这套主界面方案的本质是：

一个基于 Cocos 的运行时动态 UI 页面，而不是依赖大量静态美术拼装的页面。

对应落地方向应统一为：

结构：单主区 + 左右可收缩卡片
适配：视口分类 + 尺寸令牌 + 最大宽度限制
实现：组件工厂动态创建 + 状态驱动 + 节点复用
资源：宠物保留美术扩展，其余 UI 优先程序生成
流程：先原型，后令牌，再视觉，最后补资源
