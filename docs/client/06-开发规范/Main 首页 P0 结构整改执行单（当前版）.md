# 四、当前真实问题（按优先级排）

## P0-1：壳层适配策略还不自然

当前页面虽然不再直接锁死 `1280x720`，但非 compact 设备仍然容易被统一按固定比例收束，结果就是：

* 大屏像中间摆了一块板子
* 竖向设备上下留白过大
* 页面不像自然长在当前舞台里

### 本轮目标

* 壳层必须先占住**当前设备的真实可用舞台**
* 再根据设备档位做合理收束
* 不允许继续出现“中间摆一块板子”的感觉

---

## P0-2：compact 仍然不够独立

当前 compact 最容易继续走偏成“桌面骨架压缩版”。

### 本轮目标

compact 必须是**独立骨架**，不是桌面布局缩小版：

* 上：left-panel 压缩信息区
* 中：center-panel 主舞台区
* 下：bottom-bar 行动区
* right-panel 不常驻
* 日志反馈改由 left-panel 内的摘要区承担

---

## P0-3：left-panel 还没完全变成真实纵向流

客户端现在最容易继续用：

* 一组固定 y
* 一组固定块高
* 一组固定间距
  来“模拟”纵向栈。

### 本轮目标

left-panel 必须真正固定成：

1. 顶部状态内容区
2. 轻量摘要区
3. 最小聊天单元

这三块必须按**容器顺序往下排**，不能继续靠 `summaryY / miniChatY / statY` 这类变量手摆。

---

## P0-4：scene 还没真正成为舞台系统

当前 scene 已经开始收口，但仍然容易退化成“一个盒子 + 一堆固定坐标零件”。

### 本轮目标

scene 必须拆成：

* `ambient layer`
* `stage layer`

并且：

* 宠物主体
* 气泡
* 名字牌
* 地面底板
* 最小必要承载元素

全部收进 scene 内，不再散挂在 center-panel 根上。

---

## P0-5：页面主次关系仍然发灰发虚

现在的问题已经不只是“高光太多”，而是：

* `AppShell / AppFrame / MainArea / BottomBar / 左中右 panel` 太像
* 容器层次不清
* 页面像罩了一层淡奶白雾

### 本轮目标

* `shell / frame / scene / mainArea / bottomBar` 用更克制的容器样式
* 只有业务卡片保留强调层次：

  * XP 卡
  * 状态卡
  * 摘要卡
  * 事件卡
* 页面不是“更轻”，而是“更清楚”
