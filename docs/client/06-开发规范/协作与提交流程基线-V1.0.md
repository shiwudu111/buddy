## 适用范围
- `buddy-client`
- WSL 开发区 / Windows 审核区 / GitHub 远端协作流程

## 协作模型

```text
WSL 开发区 -> 本地 commit -> 同步到 Windows 审核区 -> 审核通过 -> Windows push
```

## 核心规则
1. **WSL 只负责开发与 commit，不直接 push 到远端。**
2. **正式发布以 Windows 审核仓库为准。**
3. **不得绕过人工审核流程。**
4. **不得手工修改 `.scene` / `.prefab` 的 JSON 内容。**
5. **遇到权限、认证、同步异常，先停手并上报，不反复试错。**

## 分支约定
- `main`：稳定分支，禁止直接提交
- `develop`：开发集成分支
- `feature/*`：功能分支
- `hotfix/*`：紧急修复分支

## 日常工作流

### 1. 开始前
```bash
git branch
git fetch origin
```

### 2. 开发中
- 小步提交，按模块提交。
- 一个提交只解决一个相对明确的问题。

### 3. 提交后
- 将工作区同步到 Windows 审核目录。
- Windows 审核通过后，再由 Windows 执行 `git push origin develop`。

### 4. 审核后同步
```bash
git fetch origin
```

## 禁止事项
- 禁止在 WSL 中直接 `git push`
- 禁止在状态不清楚时自行 `reset --hard`
- 禁止绕过审核直接发布
- 禁止提交 `.env`、日志、缓存、临时文件
- 禁止把“复制文件后重新提交”当作保留原始 commit SHA 的方式

## 冲突处理
1. `git stash`
2. `git fetch`
3. 明确来源后再 `merge`
4. `git stash pop`
5. 手动解决冲突
6. 重新提交并再次送审

## 提交格式

```text
<type>(<scope>): <subject>
```

## 推荐类型
| type | 含义 |
|------|------|
| feat | 新功能 |
| fix | 修复问题 |
| docs | 文档更新 |
| style | 样式或格式调整 |
| refactor | 重构 |
| test | 测试 |
| chore | 工具/构建/杂项 |
| perf | 性能优化 |

## 推荐 scope
| scope | 适用范围 |
|------|------|
| client | 客户端整体 |
| login | 登录模块 |
| main | 主场景 |
| homework | 作业模块 |
| scene | 场景文件 |
| ui | UI 结构 |
| network | 网络请求 |
| storage | 本地存储 |
| docs | 文档 |

## 提交示例
- `feat(login): 添加注册按钮联调逻辑`
- `fix(main): 修复宠物状态刷新失败`
- `docs(api): 更新 MVP 接口基线`

## 提交要求
- 主题尽量简短、明确、可追踪
- 默认使用中文描述
- 不把多个不相关改动塞进同一个提交

## 关于 commit SHA
- 默认情况下：
  - **WSL 本地 SHA** 只代表开发侧记录
  - **Windows 审核仓库 SHA** 才是正式历史
- 若确实需要保留原始 SHA，使用：
  - `git format-patch` + `git am`
  - `git bundle` + `git fetch`
