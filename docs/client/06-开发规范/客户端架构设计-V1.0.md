## 文档定位
- **编写时间**：2026-04-01
- **适用阶段**：MVP 前期开发 / 接手重构
- **适用仓库**：`buddy-client`
- **设计依据**：
  - `05-决策记录/DECISION-RESULT-FROZEN-V1.0.md`
  - `03-API接口/API-MVP-BASELINE-V1.0.md`
  - `04-数据库/DATABASE-ER.md`

## 目标
本次客户端重构不追求一次性做成最终产品，而是建立一套 **可联调、可扩展、可持续开发** 的 MVP 客户端框架，解决以下问题：

1. 现有客户端过度依赖单场景脚本，网络、状态、UI 混在一起。
2. `Main.scene` 基本为空壳，原有代码只能算最小联调页，无法支撑完整 MVP。
3. `Homework.scene` 当前不稳定，不能把后续开发建立在手工场景资产之上。
4. 家长端、作业中心、宠物状态联动缺少统一状态与服务层。

## 重构原则

### 1. 保留合理资产
- 保留现有场景名：`Login` / `Main`
- 保留现有持久化键：`token` / `user` / `petId`
- 保留当前最小联调逻辑中已正确的部分：
  - 登录态持久化
  - 统一 API baseUrl
  - `Main` 作为主入口场景

### 2. 不依赖手工场景重做
- 登录场景继续复用已有输入框节点。
- 主界面改为 **运行时构建容器式 UI**，避免卡在资源编辑和 scene JSON 手改上。
- 作业中心并入 `Main` 的子视图，减少额外场景维护成本。

### 3. 先补框架，再扩业务
- 页面脚本只负责：
  - 用户交互
  - 场景渲染
  - 调用服务层
- 服务层负责：
  - 联调 API
  - 状态落地
  - 数据归一化
- 状态层负责：
  - 当前用户
  - 当前宠物
  - 作业历史
  - 今日作业状态

## 分层说明

### 1. `core`
基础配置与存储，不承载业务含义。

职责：
- 环境配置
- 本地存储
- 常量

### 2. `domain`
领域枚举和跨模块共享模型。

职责：
- 场景名
- 作业科目
- 主界面页签

### 3. `network`
原始接口访问层。

职责：
- 发请求
- 鉴权头注入
- 通用错误兜底
- API 方法定义

不负责：
- 业务拼装
- 场景跳转
- 页面提示

### 4. `services`
业务服务层，是客户端 MVP 的关键骨架。

职责：
- 登录 / 注册 / 会话恢复
- 宠物创建 / 刷新 / 喂养
- 作业提交 / 历史 / 今日状态
- 家长绑定 / 孩子总览 / 周报
- 数据写入 `AppState`

### 5. `app`
应用运行态状态中心。

当前统一维护：
- `currentUser`
- `currentPet`
- `homeworkHistory`
- `todayHomeworkStatus`
- `petId`

### 6. `navigation`
场景切换入口统一封装，避免页面里到处直接 `director.loadScene(...)`

### 7. `ui/common`
通用页面基类与运行时 UI 构建器。

当前设计理由：
- `Main.scene` 为空，必须支持代码构建页面
- 不依赖手工 prefab / scene 才能快速推进 MVP

### 8. `ui/login` / `ui/main` / `ui/homework`
页面层只关心“页面逻辑 + 调服务 + 渲染结果”。

## 页面结构设计

### 1. Login
目标：
- 支持登录
- 支持注册
- 支持已登录态恢复

特点：
- 保留现有输入框节点
- 若场景中缺少注册按钮，运行时补建

### 2. Main
`Main` 是整个 MVP 的主容器场景。

按角色区分：
- `CHILD`
  - 宠物总览
  - 作业中心
- `PARENT`
  - 绑定孩子
  - 孩子宠物状态
  - 周报留壳显示

按学生端页签区分：
- `overview`
- `homework`

### 3. Homework
当前不作为主流程独立入口，保留为兜底说明页。

原因：
- 现阶段把作业中心并入 `Main` 更稳定
- 避免依赖额外场景资产

## 状态流设计

### 启动流
1. Login 启动
2. 若本地有 token，则调用 `/auth/me`
3. 成功则恢复 `currentUser` 并进入 `Main`
4. 失败则清理本地会话并回到登录态

### 学生端主链路
1. 登录 / 注册
2. 进入 `Main`
3. 刷新宠物状态
4. 查看今日作业状态 / 作业历史
5. 创建宠物或喂养宠物
6. 进入作业中心提交作业
7. 刷新作业记录与宠物状态

### 家长端主链路
1. 登录
2. 进入 `Main`
3. 输入孩子标识进行绑定
4. 调用孩子宠物总览接口
5. 调用周报留壳接口

## 为什么选择运行时 UI

### 优点
- 不阻塞开发
- 不依赖设计资源先到位
- 方便统一视觉和布局逻辑
- 可直接验证接口和状态联动

### 代价
- 短期 UI 观感不如专门设计的 scene / prefab
- 后期接设计稿时需要逐步替换成 prefab 化结构

### 结论
对当前 MVP 阶段，这是正确取舍。

## 后续演进建议

### 第一步：稳定联调
- 优先保证登录、宠物、作业、家长四条链路跑通

### 第二步：替换运行时 UI
- 设计资源稳定后，逐步把运行时 UI 替换为 prefab / scene 配置
- 服务层与状态层无需推翻，只替换页面实现

### 第三步：拆模块
- 把 `Main` 中学生端与家长端进一步拆成独立面板组件
- 引入更清晰的 UI 组件库

## 现实困难与限制

1. `Main.scene` 当前为空，无法直接承接完整页面，只能由代码构建主容器。
2. `Homework.scene` 当前不稳定存在，因此作业中心必须并入 `Main`。
3. 客户端已按新基线整理，但后端仍可能存在字段命名漂移，需要联调时逐项确认。
4. 文件上传仍是 MVP 简化区，当前客户端以文本/URL 方式先完成联调，不等于最终素材上传方案。

## 结论
这套框架的目标不是“做出最终 UI”，而是把客户端从联调脚本提升为真正可维护的 MVP 客户端骨架。

它已经具备：
- 基础分层
- 主链路容器
- 角色区分
- 状态集中管理
- 服务层与接口层解耦
- 后续替换 UI 资源而不推翻业务代码的能力
