## 背景
学生端 `Main` 正在从“文档式宠物状态页”升级为更接近宠物主页的主界面。当前这轮不仅有视觉重做，还明确包含新的真实联动能力：
- 口粮库存
- 作业产出口粮
- 选粮喂养
- `energy / health` 宠物状态字段

这意味着后端不能只“顺手补接口”，而要先冻结契约，再进入前后端并行开发。

## 本轮冻结口径
### 1. 宠物字段
正式对外新增：
- `energy`
- `health`

现有字段继续保留并沿用：
- `pet_id`
- `name`
- `level`
- `hunger`
- `mood`
- `experience`
- `stage`
- `status`

### 2. 口粮系统
口粮类型建议冻结为：
- `xp`
- `energy`

口粮品质建议冻结为：
- `normal`
- `premium`
- `advanced`

### 3. 作业奖励
作业提交成功后，后端结算口粮奖励，前端不扫描历史作业，不回放历史事件。

### 4. 喂养规则
每次喂养消耗 1 份口粮；库存不足时必须返回稳定失败语义。

### 5. 质量映射
默认分数映射：
- `0-59 -> normal`
- `60-84 -> premium`
- `85-100 -> advanced`

## 后端任务拆分
### Task A：冻结资源系统口径
- 明确 `energy / health` 是否作为正式字段返回。
- 明确口粮库存模型。
- 明确作业奖励规则。
- 明确分数到品质的映射。
- 明确喂养请求体与返回体。
- 明确库存不足时只靠库存判断，不额外让前端猜。

### Task B：扩展宠物状态读取
- 学生端主页刷新时一次拿到宠物状态与口粮库存。
- 推荐先做聚合返回，减少主页首屏拼装复杂度。

### Task C：作业提交奖励口粮
- `POST /api/v1/homeworks/submit` 继续作为提交入口。
- 后端在成功响应里返回本次奖励结果与最新宠物状态。

### Task D：选粮喂养
- 喂养接口支持明确选择口粮。
- 后端负责校验库存、扣减库存、更新宠物状态、返回最新库存。

### Task E：兼容与回归
- 保证登录、学生端主链路、作业中心、聊天、家长端不被新资源系统破坏。
- 若旧接口继续保留，要标清“仍有效 / 兼容 / 替代中”。

## 联调规则
### 1. 冻结顺序
- 先冻结字段和接口，再开始前后端并行开发。
- 冻结项至少包括：
  - 新字段名
  - 新接口路径
  - 请求体
  - 响应体
  - 分数到品质映射
  - 学科到口粮类型映射
  - 错误码和错误文案语义

### 2. 单一真相文档
- 本文是本阶段主契约。
- 旧 `后端协作需求-客户端联调-V1.0.md` 只保留历史背景，不再作为资源系统主契约。
- 若三者冲突，以本文和 `CURRENT-SOURCE-OF-TRUTH.md` 为准。

### 3. 前后端职责边界
- 后端负责：
  - 口粮库存真值
  - 奖励与消耗结算
  - 新宠物状态字段
  - 喂养合法性判断
- 前端负责：
  - 主界面视觉还原
  - 缺粮提示与跳作业中心
  - 选粮弹框
  - 最近日志展示
- 前端不负责：
  - 伪造库存
  - 伪造分数映射结果
  - 猜测经验 / 体力 / 健康度公式

### 4. 联调节奏
- 第一步：后端先交静态契约
- 第二步：前端按冻结契约改类型与主界面
- 第三步：后端提供可调环境
- 第四步：按固定用例联调
- 第五步：通过后再更新验收清单和当前状态文档

### 5. 环境与数据
- 至少准备 4 类固定测试状态：
  - 有宠物、有库存
  - 有宠物、无库存
  - 刚提交作业可获得口粮
  - 喂养后会触发升级或阶段变化
- 这些数据必须写进固定联调测试数据表，不能只靠口头说明。

### 6. 变更规则
- 联调期间若后端要改字段、路径、返回结构：
  - 必须先改文档
  - 再通知前端
  - 再一起改代码
- 禁止“环境已经变了，但文档没变”。

## 验收与阻塞规则
### 后端通过标准
- 主界面可稳定读取新宠物状态与库存
- 作业提交后能稳定产出口粮
- 选粮喂养可成功扣减库存并返回最新状态
- 库存不足时返回稳定失败语义
- 不破坏现有聊天、作业历史、家长端

### 前后端联合验收场景
- 学生端登录进入主页，可看到宠物状态与库存
- 点击喂养：
  - 有库存时可选粮并成功喂养
  - 无库存时弹缺粮提示
  - 点击 `提交作业` 可进入作业中心
- 作业提交成功后回主页，可看到口粮变化与日志更新
- 喂养后主页左侧状态区和宠物成长页状态一致
- 聊天仍可正常发送，不受新资源系统影响

### 停止条件
出现以下任一情况时，本轮实现应先停止并记录阻塞，不扩展范围：
- 后端尚未冻结 `energy / health / foods` 字段
- 喂养接口还不能表达“选择哪种口粮”
- 作业提交奖励规则仍在频繁变动
- 后端不能提供固定联调数据
- 前端开始需要自己伪造库存或结算逻辑才能继续

## 假设
- 这轮真实资源系统只服务学生端主界面，不扩到家长端展示。
- `一起玩` 本轮不是后端任务，不需要配套接口。
- 右侧日志区本轮以前端事件流为主，不额外要求后端活动流接口。
- 如果后端本轮来不及交付完整资源系统，应拆成两阶段：
  - 阶段 1：先完成主界面视觉还原与交互占位
  - 阶段 2：再接真实口粮系统
