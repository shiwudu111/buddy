## 一、为什么需要提交规范

### 1. 团队协作价值
- **可读性**：清晰的提交信息，便于理解代码变更
- **可追溯性**：方便查找特定类型的变更
- **自动化**：支持自动化生成CHANGELOG
- **规范化**：统一团队提交习惯，减少沟通成本

### 2. 项目维护价值
- **版本管理**：清晰的提交历史，便于版本回退
- **代码审查**：明确的变更类型，提高审查效率
- **问题追踪**：与Issue/Bug关联，便于问题追踪
- **质量保证**：规范的提交促进代码质量

## 二、提交格式规范

### 1. 基本格式
```
<type>(<scope>): <subject>

<body>

<footer>
```

### 2. 各部分组成

#### 类型（Type） - 必填
描述提交的变更类型：

| 类型 | 说明 | 示例 |
|------|------|------|
| **feat** | 新功能 | `feat(auth): 添加用户登录功能` |
| **fix** | 修复Bug | `fix(api): 修复用户注册接口500错误` |
| **docs** | 文档更新 | `docs(readme): 更新项目启动说明` |
| **style** | 代码格式调整 | `style(ui): 调整按钮样式间距` |
| **refactor** | 代码重构 | `refactor(db): 重构数据库连接池` |
| **test** | 测试相关 | `test(auth): 添加登录功能单元测试` |
| **chore** | 构建过程或辅助工具变动 | `chore(deps): 更新axios到最新版本` |
| **perf** | 性能优化 | `perf(api): 优化接口响应时间` |
| **ci** | CI/CD相关 | `ci(github): 添加自动化测试流水线` |
| **build** | 构建系统或外部依赖 | `build(webpack): 优化打包配置` |
| **revert** | 回退提交 | `revert: 回退错误的数据库迁移` |

#### 范围（Scope） - 可选
描述提交影响的范围，通常是模块或文件：

| 范围 | 说明 | 示例 |
|------|------|------|
| **auth** | 认证授权模块 | `feat(auth): 添加JWT token刷新` |
| **api** | API接口模块 | `fix(api): 修复用户信息接口` |
| **db** | 数据库相关 | `refactor(db): 优化查询性能` |
| **ui** | 用户界面 | `style(ui): 调整登录页面布局` |
| **config** | 配置相关 | `chore(config): 更新环境变量配置` |
| **test** | 测试相关 | `test(api): 添加接口测试用例` |
| **deps** | 依赖更新 | `chore(deps): 更新项目依赖` |

#### 主题（Subject） - 必填
简洁的描述，不超过50个字符：
- 使用祈使句，现在时态
- 首字母小写，结尾不要句号
- 清晰描述做了什么，而不是为什么做

**正确示例**：
- `feat(auth): 添加用户注册功能`
- `fix(api): 修复分页查询参数错误`

**错误示例**：
- `feat(auth): 添加了用户注册功能。`（有句号）
- `fix(api): 修复bug`（描述不清晰）
- `feat: 新功能`（缺少scope，描述太简单）

#### 正文（Body） - 可选
详细描述变更内容：
- 说明变更的动机和背景
- 与之前行为的对比
- 如果是破坏性变更，需要说明

**格式要求**：
- 每行不超过72个字符
- 使用空行与主题分隔
- 使用列表描述多个变更点

**示例**：
```
feat(auth): 添加第三方登录支持

- 支持微信扫码登录
- 支持GitHub OAuth登录
- 添加第三方登录配置管理
- 更新用户表结构，添加第三方ID字段

第三方登录可提升用户体验，减少注册门槛。
```

#### 页脚（Footer） - 可选
关联Issue或Breaking Changes：

**关联Issue**：
```
Closes #123
Fixes #456
Resolves #789
```

**破坏性变更**：
```
BREAKING CHANGE: 数据库表结构变更，需要迁移数据

迁移步骤：
1. 备份现有数据
2. 运行迁移脚本
3. 验证数据完整性
```

## 三、提交示例

### 示例1：新功能提交
```
feat(pet): 添加宠物喂养功能

- 实现宠物喂养接口 POST /api/v1/pets/:id/feed
- 添加饱食度计算逻辑
- 更新宠物状态变更记录
- 添加喂养次数限制（每天最多5次）

Closes #45
```

### 示例2：Bug修复
```
fix(homework): 修复作业提交时间校验错误

- 修复时区处理问题，使用UTC时间
- 修复提交时间晚于截止时间的校验逻辑
- 添加时间格式验证

原问题：作业提交时间显示错误，时区未正确处理。
解决方案：统一使用UTC时间存储和比较。

Fixes #78
```

### 示例3：文档更新
```
docs(readme): 更新项目启动文档

- 添加环境要求说明
- 更新数据库配置步骤
- 添加API接口文档链接
- 补充常见问题解答

项目启动文档需要保持最新，方便新成员快速上手。
```

### 示例4：代码重构
```
refactor(db): 重构数据库连接管理

- 提取数据库连接配置到独立模块
- 添加连接池配置，提升性能
- 统一错误处理机制
- 添加连接状态监控

BREAKING CHANGE: 数据库连接配置方式变更

迁移步骤：
1. 更新.env文件中的数据库配置
2. 运行数据库连接测试
3. 验证现有功能正常
```

### 示例5：依赖更新
```
chore(deps): 更新项目依赖版本

- axios: ^1.3.0 → ^1.6.0
- express: ^4.18.0 → ^4.19.0
- typescript: ^5.0.0 → ^5.3.0
- jest: ^29.0.0 → ^29.7.0

安全更新和性能优化。
```

## 四、特殊情况处理

### 1. 多个变更类型
如果一次提交包含多个类型的变更，以主要变更类型为准：

**正确**：
```
feat(auth): 添加新功能并修复相关问题

- 添加用户注册功能（主要）
- 修复登录页面样式问题（次要）
- 更新相关文档
```

**避免**：混合不相关的多个变更

### 2. 范围不确定
如果变更影响多个模块或范围不确定：

**使用通用范围**：
```
feat: 添加项目基础配置
fix: 修复多个模块的样式问题
chore: 更新项目依赖
```

### 3. 提交信息长度
- **主题**：不超过50字符
- **正文每行**：不超过72字符
- **总长度**：建议不超过200字符（GitHub显示限制）

### 4. 中文 vs 英文
**团队约定**：使用中文，便于团队理解

**原因**：
- 团队母语为中文，沟通效率高
- 项目文档和注释使用中文
- 减少语言切换成本

## 五、工具支持

### 1. Git Hook配置
在项目中添加commit-msg钩子，自动验证提交信息格式：

```bash
# .husky/commit-msg
#!/bin/sh
. "$(dirname "$0")/_/husky.sh"

npx --no -- commitlint --edit "$1"
```

### 2. Commitizen（可选）
交互式提交工具，引导填写规范提交信息：

```bash
# 安装
npm install -g commitizen

# 使用
git cz
```

### 3. Commitlint配置
提交信息校验规则：

```javascript
// commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [
      2,
      'always',
      ['feat', 'fix', 'docs', 'style', 'refactor', 'test', 'chore', 'perf', 'ci', 'build', 'revert']
    ],
    'subject-case': [0], // 不限制大小写
    'header-max-length': [2, 'always', 100] // 主题最大长度
  }
};
```

## 六、团队培训材料

### 1. 快速记忆口诀
```
功能新增用feat，修复Bug用fix
文档更新用docs，代码格式style
重构代码refactor，测试相关test
构建工具chore记，性能优化perf
CI/CD用ci，构建系统build
回退提交revert，范围可选记心里
```

### 2. 常见错误避免
- ❌ 提交信息太简单：`update`、`fix bug`
- ❌ 使用过去时：`fixed`、`added`
- ❌ 缺少类型：`用户登录功能`
- ❌ 主题太长：超过50字符
- ❌ 中英文混合：`fix(用户): 修复user登录问题`

### 3. 检查清单
提交前自检：
- [ ] 类型是否正确？
- [ ] 范围是否明确？
- [ ] 主题是否简洁清晰？
- [ ] 正文是否必要？如需，是否详细？
- [ ] 是否关联了相关Issue？
- [ ] 是否有破坏性变更需要说明？

## 九、附录

### 1. 类型详细说明

#### feat（新功能）
- 新增用户可见的功能
- 新增API接口
- 新增页面或组件
- 新增配置选项

#### fix（Bug修复）
- 修复功能缺陷
- 修复安全漏洞
- 修复性能问题
- 修复兼容性问题

#### docs（文档）
- 更新README
- 更新API文档
- 更新代码注释
- 更新项目文档

#### style（代码格式）
- 代码格式化
- 空格、缩进调整
- 分号、引号统一
- 不影响代码功能的样式调整

#### refactor（代码重构）
- 代码结构优化
- 设计模式应用
- 性能优化（非Bug修复）
- 代码清理和整理
