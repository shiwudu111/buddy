# 登录入口重构状态机草案-V1.0

## 文档定位
- **编写时间**：2026-04-10
- **适用仓库**：`buddy-client`
- **适用范围**：`Login.scene` 入口流程重构
- **文档状态**：草案，待联合评审确认后进入正式基线

## 一、目标
本草案用于冻结登录入口的流程状态，而不是先冻结具体视觉实现。

本次重构的目标不是“优化旧登录页样式”，而是将当前单页表单入口改造为多步骤入口流程：

`restore -> brandEntry -> roleSelect -> authForm`

这样可以将以下三个决策拆开：
1. 当前是否已有有效登录态
2. 当前用户想以哪种身份进入
3. 当前用户要执行登录还是注册

## 二、状态机范围
本状态机只负责 Login 模块内部流程，不改变当前场景边界：
- 保持 `Login -> Main`
- 第一阶段不拆分多个 Cocos Scene
- 第一阶段不改 `Main.scene` 内部逻辑

## 三、核心状态定义

### 1. `restore`
含义：
- 页面启动后的会话恢复检查状态

进入时机：
- `Login.scene` 首次加载

处理逻辑：
- 检查本地是否存在有效会话信息
- 若存在，尝试恢复登录态
- 若恢复成功，回到 `brandEntry`，并带上可续用账号
- 若无会话或恢复失败，进入 `brandEntry`

结果语义要求：
- `restoreSuccess`：恢复成功，可继续进入
- `restoreNoSession`：本地无会话，不是错误，进入 `brandEntry`
- `restoreFailure`：存在会话但恢复失败，进入 `brandEntry`

说明：
- `restoreNoSession` 不能再复用旧的 `idle`
- `idle` 属于旧单页表单语义，不再适合作为新入口流程的恢复结果
- 当 `/auth/me` 返回 `401`、token 无效或服务端明确判定当前会话不可恢复时，客户端统一按 `restoreNoSession` 处理
- `restoreFailure` 保留给网络异常、服务异常或客户端无法完成恢复流程的情形

### 2. `brandEntry`
含义：
- 品牌入口页

界面特征：
- 展示 Logo、背景氛围、一个明确主动作“开始使用”
- 展示默认账号入口条，可切出轻量账号选择弹层

允许动作：
- 若当前选中默认账号且存在可恢复登录态，点击“开始使用”直接续入 `Main`
- 若当前选中默认账号但无可恢复登录态，点击“开始使用”进入 `authForm(role=defaultAccount.role, mode=login)`
- 若当前选择“其他账号”，点击“开始使用”进入 `roleSelect`
- 点击账号入口条打开账号选择弹层
- 在弹层中切换“默认账号 / 其他账号”
- 点击弹层空白处关闭弹层

禁止事项：
- 不出现账号密码输入框
- 不出现学生/家长/登录/注册多个并列主按钮

### 3. `roleSelect`
含义：
- 身份选择页或身份选择弹层

界面特征：
- 只提供两个明确主入口：
  - 我是学生
  - 我是家长

默认规则：
- 从 `roleSelect` 进入 `authForm` 时，默认进入 `mode=login`
- 注册通过 `authForm` 内的显式切换入口完成

补充规则：
- 若后续产品决定“首次用户默认注册、回流用户默认登录”，必须在正式版中补充识别依据
- 在识别依据未冻结前，统一采用“默认登录”规则

允许动作：
- 选择学生：进入 `authForm(role=CHILD, mode=login)`
- 选择家长：进入 `authForm(role=PARENT, mode=login)`
- 返回：回到 `brandEntry`

禁止事项：
- 不在此页出现输入框
- 不在此页出现复杂错误提示与表单校验

### 4. `authForm`
含义：
- 统一认证表单页

状态参数：
- `role`: `CHILD | PARENT`
- `mode`: `login | register`

界面特征：
- 统一表单容器
- 通过 `role` 与 `mode` 切换标题、主按钮、副按钮文案

主动作用规则：
- `mode=login` 时：
  - 主按钮：登录并进入
  - 次级文本入口：没有账号，去注册
- `mode=register` 时：
  - 主按钮：注册并进入
  - 次级文本入口：已有账号，去登录

返回规则：
- 从 `authForm` 返回时，统一回到 `brandEntry`
- `roleSelect` 只作为进入 `authForm` 前的一次分流，不再作为表单返回落点

错误提示规则：
- 错误提示必须就近显示在输入框下方或表单区内部
- 不再放在页面底部

登录角色一致性规则：
- 登录请求不以 `role` 作为接口参数
- `role` 在登录态下只用于界面文案与流程上下文，不作为前端强校验条件
- 如果用户在 `roleSelect` 选择了学生，但输入了家长账号：
  - 允许提交
  - 以后端返回的真实 `role` 为准
  - 登录成功后的跳转、会话写入、埋点身份都以真实 `role` 为准
- 前端不在第一阶段限制“所选身份必须与账号真实 role 一致”

注册角色规则：
- 注册请求显式携带 `role`
- 学生注册传 `CHILD`
- 家长注册传 `PARENT`

注册失败语义要求：
- 用户名已存在：需要返回稳定、可直接展示的错误信息
- 参数缺失或参数不合法：需要返回稳定、可直接展示的错误信息
- 未授权或非法 token：保持与当前后端语义一致，客户端按失败提示处理

## 四、事件与迁移规则

### 启动阶段
1. `Login.scene loaded`
   - 进入 `restore`

2. `restore success`
   - `restore -> brandEntry`
   - 带上可续用账号，等待用户点击“开始使用”

3. `restore no session`
   - `restore -> brandEntry`

4. `restore failed`
   - `restore -> brandEntry`

### 品牌入口阶段
1. `click start`
   - 若当前为“其他账号”：`brandEntry -> roleSelect`
   - 若当前为“默认账号”：`brandEntry -> authForm(role=defaultAccount.role, mode=login)`
   - 若当前默认账号同时带有可恢复登录态：`brandEntry -> Main`

2. `click account entry`
   - 打开账号选择弹层
   - 不改变主状态机主路径

### 身份选择阶段
1. `select CHILD`
   - `roleSelect -> authForm(role=CHILD, mode=login)`

2. `select PARENT`
   - `roleSelect -> authForm(role=PARENT, mode=login)`

3. `click back`
   - `roleSelect -> brandEntry`

### 认证表单阶段
1. `switch to register`
   - `authForm(role=X, mode=login) -> authForm(role=X, mode=register)`

2. `switch to login`
   - `authForm(role=X, mode=register) -> authForm(role=X, mode=login)`

3. `submit login success`
   - `authForm -> Main`
   - 实际进入哪个端以后端返回的真实 `role` 为准

4. `submit register success`
   - `authForm -> Main`
   - 进入与注册 `role` 对应的端

5. `submit failure`
   - 保持在当前 `authForm`
   - 更新错误提示和状态提示

6. `click back`
   - `authForm -> brandEntry`

## 五、状态字段建议
建议新增统一流程状态，而不是继续在控制器里堆布尔值。

推荐结构：

```ts
type LoginFlowStep = "restore" | "brandEntry" | "roleSelect" | "authForm";
type LoginFlowRole = "CHILD" | "PARENT" | null;
type LoginFlowMode = "login" | "register";

type LoginFlowState = {
  step: LoginFlowStep;
  role: LoginFlowRole;
  mode: LoginFlowMode;
};
```

认证结果建议同步显式化：

```ts
type RestoreResult =
  | { kind: "restoreSuccess" }
  | { kind: "restoreNoSession" }
  | { kind: "restoreFailure"; message: string };
```

说明：
- `LoginAuthCoordinator` 继续只负责认证动作
- `LoginFlowState` 负责页面阶段与入口流程
- `restoreNoSession` 必须是独立结果，不再复用旧 `idle`

## 六、第一阶段不做的事
- 不拆多 Scene
- 不引入复杂账号管理流程
- 不在此阶段决定多账号体系
- 不在此阶段新增后端业务接口
- 不在此阶段改 `Main.scene` 逻辑

## 七、与现有方案的冲突点
本草案与当前旧登录页方案的直接冲突点如下：
1. 不再保留“账号 + 密码 + 登录 + 学生注册 + 家长注册”同屏结构
2. `Login.scene` 首屏不再直接显示表单
3. 登录与注册不再是首屏并列动作，而是认证表单内的模式切换
4. 登录页固化方案需要从“单页表单固化”改成“多容器流程固化”

## 八、验收标准
满足以下条件，可认为状态机方案落地成功：
1. 首屏只有一个明确主动作“开始使用”
2. 首屏提供默认账号 / 其他账号的轻量入口，不再让用户直接面对旧同屏表单
3. 首次用户不会在同一屏同时面对身份判断与表单输入
4. 学生与家长在身份选择阶段即完成明确分流
5. `roleSelect` 进入 `authForm` 时默认是登录模式
6. 登录与注册共用同一套表单容器，但模式清晰
7. 登录成功时以后端返回的真实 `role` 为准完成跳转
8. `restore` 无会话、恢复失败与恢复成功都能稳定回到 `brandEntry`
9. 认证表单返回时统一回到账号入口
10. 新流程不改变当前 `Login -> Main` 场景边界

## 九、下一步建议
1. 在本草案通过后，输出低保真线框图
2. 基于本草案编写客户端实施方案
3. 将 `restoreNoSession` / `restoreFailure` 的结果语义同步到认证协调层
4. 将 `/auth/me` 的 `401 / token 无效 -> restoreNoSession` 联调口径写入后续基线文档
5. 将注册失败错误语义写入后续联调文档，确保表单页可稳定做就近提示
6. 后续再统一更新 UI 固化方案与联调文档中的登录入口部分
