# 登录入口重构客户端实施方案草案-V1.0

## 文档定位
- **编写时间**：2026-04-10
- **适用仓库**：`buddy-client`
- **适用范围**：`Login.scene` 登录入口流程重构
- **文档状态**：草案，待联合评审确认后进入正式实施

## 一、背景与目标
本方案用于承接《登录入口重构状态机草案-V1.0》，将现有“单页表单登录”改造成“多步骤入口流程”。

本次工作的目标不是简单替换登录页视觉，而是完成以下三项产品层调整：
1. 将“是否已有登录态”从首屏表单中抽离
2. 将“用户身份选择”从登录/注册行为中抽离
3. 将“登录/注册表单输入”收敛到统一认证表单层

第一阶段推荐采用同一 `Login.scene` 内多容器流程，不修改当前 `Login -> Main` 路由边界，不在第一阶段拆分多场景。

## 二、实施原则
1. **先冻结流程，后冻结实现**
   - 先以状态机统一入口流程，再决定视觉与节点细节。
2. **先改流程边界，不先改场景路由**
   - 第一阶段保留 `Login.scene` 作为唯一登录入口场景。
3. **认证动作与页面流程分离**
   - `LoginAuthCoordinator` 继续只负责恢复登录态、登录、注册。
   - 新增流程状态层，负责页面阶段、身份选择、模式切换和返回规则。
4. **默认优先老用户效率**
   - `roleSelect -> authForm` 第一阶段默认进入 `mode=login`。
   - 注册通过表单内显式切换完成。
5. **登录成功以真实角色为准**
   - 登录接口不依赖前端选择的 `role`。
   - 登录后的跳转、状态与埋点以服务端返回的真实 `role` 为准。
6. **恢复结果语义显式化**
   - `restoreNoSession` 与 `restoreFailure` 必须成为独立结果，不再复用旧 `idle`。
7. **先跑通主链路，再做视觉精修**
   - 第一阶段先追求可联调、可回归，不引入额外复杂能力。

## 三、目标流程
第一阶段正式目标流程如下：

```text
restore -> brandEntry -> roleSelect -> authForm(role, mode)
```

默认规则：
- `restore` 无会话或恢复失败时，统一回到 `brandEntry`
- `roleSelect` 进入 `authForm` 时默认是 `mode=login`
- `authForm` 中通过显式切换进入 `mode=register`

## 四、客户端改造范围

### 1. 保留并继续使用的能力
- `Login.scene` 作为登录入口场景
- `SceneRouter` 现有 `Login -> Main` 路由
- `LoginAuthCoordinator` 中的恢复登录态、登录、注册能力
- `AuthService` 现有登录与注册接口封装

### 2. 需要新增的能力
- `LoginFlowState` 或同级流程协调层
- 登录入口页面阶段切换能力
- 默认账号与其他账号的轻量入口切换能力
- 身份选择与表单模式切换能力
- 页面返回规则与状态恢复规则

### 3. 需要改造的能力
- `LoginController`
  - 从“单页表单控制器”改为“页面入口编排控制器”
  - 不再直接持有单一表单页假设
- `LoginViewOrchestrator`
  - 从“单表单面板编排器”改为“多容器视图编排器”
- `LoginSceneStructure`
  - 从“单登录表单结构”改为“入口层 / 身份层 / 表单层”骨架结构
- `LoginVisualStyler`
  - 从“单页控件装饰”扩展为分容器样式支持

## 五、推荐代码结构

### 1. 流程层
建议新增：
- `LoginFlowState.ts`
  - 定义 `step / role / mode`
- `LoginFlowCoordinator.ts`
  - 负责状态迁移、事件响应、返回规则

建议保留：
- `LoginAuthCoordinator.ts`
  - 仅处理认证动作与认证结果

职责边界建议：
- `LoginController`
  - 绑定事件
  - 驱动 flow coordinator
  - 驱动 auth coordinator
  - 根据状态结果刷新 UI
- `LoginFlowCoordinator`
  - 不直接发登录请求
  - 不直接依赖后端接口
  - 只负责“当前在哪一页”和“下一步去哪”
- `LoginAuthCoordinator`
  - 不处理页面跳转细节
  - 不持有品牌入口和身份页状态

### 2. 认证结果层
建议将恢复会话结果显式化，避免旧表单语义泄漏到新流程：

```ts
type RestoreResult =
  | { kind: "restoreSuccess" }
  | { kind: "restoreNoSession" }
  | { kind: "restoreFailure"; message: string };
```

建议同步保留登录/注册结果中的真实用户信息，以便登录后以服务端 `role` 为准：

```ts
type AuthSubmitResult = {
  ok: boolean;
  role?: "CHILD" | "PARENT";
  message?: string;
};
```

说明：
- 第一阶段不要求大改后端接口
- 但协调层必须提供足够清晰的结果语义，不能继续让控制器猜测旧 `idle` 的含义

### 3. 视图层
建议将 `Login.scene` 内部组织为三个主容器：
- `BrandEntryLayer`
- `RoleSelectLayer`
- `AuthFormLayer`

说明：
- 第一阶段只要求逻辑上分层，不要求拆成多个 scene
- 各层之间通过显示/隐藏或激活/停用切换
- 视觉文档中的动画和装饰元素可在主链路跑通后逐步补入

## 六、关键口径

### 1. roleSelect 默认进入哪种模式
第一阶段统一采用：
- `roleSelect -> authForm(role=X, mode=login)`

原因：
- 优先照顾无登录态老用户与回流用户效率
- 避免所有用户都先被推向注册态

补充说明：
- 如果后续产品希望“首次用户默认注册、回流用户默认登录”，必须先定义识别依据
- 在识别依据冻结前，不采用条件分流实现

### 2. 登录时若所选身份与账号真实角色不一致，以什么为准
第一阶段统一采用：
- 允许提交
- 以后端返回的真实 `role` 为准

前端处理规则：
- 所选 `role` 仅用于进入对应文案与上下文
- 登录成功后的跳转、会话信息和埋点身份均以真实 `role` 为准
- 不在第一阶段引入“前端强校验身份一致性”

### 3. restore skipped / failed 返回什么结果
第一阶段统一采用：
- `restoreSuccess`
- `restoreNoSession`
- `restoreFailure`

实施要求：
- `restoreSuccess` 表示“本地存在可续用会话”，进入 `brandEntry` 后可直接续入
- `restoreNoSession` 表示“本地没有可恢复会话”，不是错误
- `restoreFailure` 表示“本地存在会话但恢复失败”
- 三者最终都回到 `brandEntry`
- 不再复用旧 `idle`

## 七、阶段实施顺序

### 阶段 A：流程冻结
目标：
- 输出登录入口状态图
- 明确页面切换规则
- 明确学生/家长分流规则
- 明确登录/注册模式切换规则
- 明确恢复结果语义

交付物：
- 《登录入口重构状态机草案-V1.0》
- 本实施方案草案

### 阶段 B：代码结构调整
目标：
- 新增流程状态层
- 缩减 `LoginController` 职责
- 将认证动作与入口流程彻底分离
- 将恢复结果语义从旧 `idle` 中拆出

建议动作：
1. 新增 `LoginFlowState` 与 `LoginFlowCoordinator`
2. 让 `LoginController` 只做事件绑定与状态桥接
3. 为 `LoginAuthCoordinator` 补充显式恢复结果类型
4. 保持登录与注册接口尽量稳定

### 阶段 C：场景与视图改造
目标：
- 在 `Login.scene` 中引入多容器布局
- 保留现有路由和登录态恢复逻辑
- 完成品牌入口、身份选择、认证表单三层切换

建议动作：
1. 在场景内增加三层容器
2. 将旧单页表单节点迁移到 `AuthFormLayer`
3. 调整 `LoginViewOrchestrator` 与 `LoginSceneStructure`
4. 在 `authForm` 中补齐登录/注册切换入口

### 阶段 D：联调与回归
目标：
- 验证学生/家长分流、登录、注册、恢复会话
- 验证 `Login -> Main` 未回归
- 验证异常路径、返回路径与角色纠偏规则

最小验收矩阵：
1. 有效学生登录态启动后回到账号入口，并可通过默认账号直接续入 `Main`
2. 有效家长登录态启动后回到账号入口，并可通过默认账号直接续入 `Main`
3. 无登录态时先进入品牌入口页
4. 选择学生后默认进入学生登录表单
5. 选择家长后默认进入家长登录表单
6. 登录成功后按真实 `role` 进入正确端
7. 注册成功后按注册 `role` 进入正确端
8. 无会话恢复时稳定进入 `brandEntry`
9. 恢复失败时稳定进入 `brandEntry`
10. 登录/注册失败时停留在当前表单并显示错误
11. 认证表单点击返回时，统一回到账号入口页

### 阶段 E：文档与基线更新
目标：
- 在新流程稳定后再统一更新基线
- 明确哪些旧方案已被替代

建议后续更新文档：
- `客户端UI固化方案-V1.0.md`
- `UI固化执行单-V1.0.md`

## 八、后端与联调影响评估

### 1. 第一阶段默认不要求后端新增接口
- `/auth/login`
- `/auth/register`
- `/auth/me`

已确认可直接支撑第一阶段客户端改造的能力：
- `POST /api/v1/auth/register` 已支持 `role=CHILD | PARENT`
- `POST /api/v1/auth/login` 成功后返回统一 `user` DTO，且包含 `role`
- `GET /api/v1/auth/me` 可用于恢复登录态

### 2. 需要确认的后端约束
- 注册接口正式支持 `role=CHILD | PARENT`
- 登录成功后的用户信息中可稳定返回 `role`
- 会话恢复在“无会话”和“恢复失败”两种情况下，客户端可获得清晰结果语义

推荐写死的联调口径：
- 登录页中用户先选“学生/家长”只是前端流程分流，真正登录成功后的身份以后端返回的 `user.role` 为准
- `/auth/me` 返回 `401`、token 无效或服务端明确判定当前会话不可恢复时，客户端统一视为“无可恢复登录态”，回到 `brandEntry`
- 注册失败时，后端继续保持稳定错误语义，至少覆盖：
  - 用户名已存在
  - 参数缺失或参数不合法
  - 未授权或非法 token

### 3. 联调口径变化
- 原来的“登录页验收”将从“单页表单验证”改为“入口流程验证”
- 学生/家长分流验证从按钮文案层升级为流程层
- 登录成功后的角色判定统一以后端返回结果为准
- 恢复登录态时，`401 / token 无效` 不再作为页面错误态停留，而是直接回到 `brandEntry`
- 恢复成功时，第一阶段也先回到 `brandEntry`，由用户通过默认账号继续进入
- 注册失败提示以稳定错误语义驱动就近表单提示

## 九、风险与控制策略

### 1. 风险：把新流程硬塞进旧单页模型
控制策略：
- 先建立流程状态层，再改视图层

### 2. 风险：默认注册导致回流用户路径变长
控制策略：
- 第一阶段统一默认登录，不默认注册

### 3. 风险：前端所选角色与真实账号角色不一致
控制策略：
- 登录时允许提交，成功后以真实 `role` 为准完成跳转和状态纠偏

### 4. 风险：恢复结果语义与旧表单状态混用
控制策略：
- 为 `restoreNoSession` 与 `restoreFailure` 提供独立结果类型

### 5. 风险：注册失败错误语义不稳定，导致表单提示漂移
控制策略：
- 在联调文档中明确错误类别与前端提示映射
- 客户端不依赖模糊自由文案猜测错误类型
### 6. 风险：改入口时连带影响主场景路由
控制策略：
- 第一阶段不动 `Login -> Main` 边界

### 7. 风险：联调口径与旧文档冲突
控制策略：
- 在新流程跑通前，不直接覆盖旧冻结文档正文
- 先产出草案与变更说明

## 十、当前不纳入第一阶段的内容
- 多 Scene 登录入口拆分
- 多账号管理系统
- 复杂账号切换
- 大规模动画演出
- `Main.scene` 结构调整
- 新增后端业务接口

## 十一、建议结论
建议将本次工作正式定义为“登录入口流程重构”，而不是“登录页视觉优化”。

客户端推荐落地顺序为：
1. 先输出状态机与实施方案
2. 再调整控制器与流程边界
3. 再落场景容器与 UI 结构
4. 再完成联调与回归
5. 最后统一更新基线文档与验收口径

如果严格按上述顺序推进，本次改造可以在不打乱现有主链路的前提下，完成从“单页表单模型”向“多步骤入口模型”的迁移。
