# Login.scene 工程落地蓝图 V1.0

## 一、落地目标

本蓝图只解决 4 件事：

1. 场景结构唯一
2. 节点命名唯一
3. 视觉层级唯一
4. 脚本绑定唯一

执行结果应满足：

- 登录页主要 UI 由场景固定承载，不依赖运行时补节点。
- `StatusLabel` 属于登录功能区内部，不是页面底部独立区。
- 火狐狸是背景氛围角色，不参与交互，不遮挡输入区。
- `LoginController` 只负责绑定、状态切换、登录提交，不负责动态拼整套 UI。

## 二、总原则

一句话：

**背景做情绪，功能做操作，结构必须唯一。**

执行时优先级固定为：

1. 可用性
2. 结构稳定
3. 视觉统一
4. 轻量动效

## 三、最终场景结构

这是推荐的唯一结构：

```text
Canvas
└── LoginPage
    ├── BackgroundLayer
    │   ├── BgGradient
    │   ├── FoxVisual
    │   └── AmbientShapes
    ├── ContentLayer
    │   ├── LogoArea
    │   │   ├── MascotIcon
    │   │   └── TitleLabel
    │   └── LoginPanel
    │       ├── UsernameInput
    │       ├── PasswordInput
    │       ├── LoginButton
    │       ├── RegisterButton
    │       ├── StatusLabel
    │       └── LoadingNode
    └── ControllerRoot
        └── LoginController
```

## 四、节点职责定义

### 1. Canvas
- 全页根节点。
- 不承担业务逻辑，只承载 UI。

### 2. LoginPage
- 登录页总容器。
- 统一控制页面布局，便于后续切场景或做整体适配。

### 3. BackgroundLayer
- 背景层，只做视觉氛围，不做交互。
- 必须在功能层下面。

### 4. ContentLayer
- 内容层，所有用户操作都在这里完成。
- 这是登录交互的核心层。

### 5. ControllerRoot
- 脚本挂载层。
- 建议把 `LoginController` 挂在独立控制节点上，避免挂在背景或按钮节点上造成职责混乱。

## 五、功能区结构写死

`LoginPanel` 必须包含以下节点，且不允许脱离：

```text
LoginPanel
 ├── UsernameInput
 ├── PasswordInput
 ├── LoginButton
 ├── RegisterButton
 ├── StatusLabel
 └── LoadingNode
```

这意味着：

- `StatusLabel` 必须在 `LoginPanel` 内部
- `LoadingNode` 必须在 `LoginPanel` 内部
- 不允许把 `StatusLabel` 单独挂到页面底部另做一层
- 不允许把 `LoadingNode` 放在全屏中央做成独立遮罩节点，除非后续需求明确升级

## 六、视觉层级蓝图

从下到上固定为：

1. `BgGradient`
2. `FoxVisual / AmbientShapes`
3. `LogoArea`
4. `LoginPanel`（输入框 / 按钮）
5. `StatusLabel / LoadingNode`

说明：

- `FoxVisual` 必须低于功能区
- `StatusLabel` 和 `LoadingNode` 必须高于按钮底板
- 背景装饰不得压到可点击元素之上

## 七、布局蓝图

### 7.1 页面分区

页面只分两个主功能区域：

- 左侧：背景视觉区
- 右侧：登录功能区

不是三段式页面。  
状态提示不是第三块独立区，而是登录功能区内部元素。

### 7.2 推荐布局关系

左侧：大面积火狐狸弱化剪影  
右侧：`LogoArea + LoginPanel`

推荐视觉关系：

- 左侧视觉区约占 `50% ~ 60%`
- 右侧功能区约占 `40% ~ 50%`

但注意，这个“占比”是视觉占比，不代表必须用硬切分栏布局。  
实际执行中更推荐：

- 火狐狸从左侧铺开到中偏左
- 登录区稳定落在中偏右

### 7.3 LoginPanel 内部顺序

从上到下固定为：

1. `UsernameInput`
2. `PasswordInput`
3. `LoginButton`
4. `RegisterButton`
5. `StatusLabel`
6. `LoadingNode`

不建议把 `StatusLabel` 插到输入框中间。  
不建议把 `LoadingNode` 放到按钮上方。  
这样可以保证操作路径符合直觉：输入 -> 提交 -> 收到反馈。

## 八、各节点工程规范

### 8.1 UsernameInput

要求：

- 节点名必须为 `UsernameInput`
- 必须挂 `EditBox`
- 必须可稳定输入
- 占位文案建议：`请输入账号`
- 不得用 `Label` 假装输入框
- 不得嵌套多套输入结构导致焦点异常

视觉规则：

- 白底
- 大圆角
- 浅蓝灰边框
- 可有用户图标，但图标不能替代占位文案

### 8.2 PasswordInput

要求：

- 节点名必须为 `PasswordInput`
- 必须挂 `EditBox`
- 必须开启密码模式
- 占位文案建议：`请输入密码`
- 与 `UsernameInput` 风格一致

视觉规则：

- 与账号框保持同宽、同高、同圆角、同边框
- 可加锁图标
- 不要单独做成更重视觉

### 8.3 LoginButton

要求：

- 节点名必须为 `LoginButton`
- 必须挂 `Button`
- 文案固定为：`登录`
- 是唯一主 CTA

视觉规则：

- 蓝青渐变
- 圆角最大
- 有轻微高光和轻微阴影
- 必须比注册按钮更有视觉重量

交互规则：

- 点击有按压回弹
- 不做夸张弹跳
- 禁止 hover 特效扩展

### 8.4 RegisterButton

要求：

- 节点名必须为 `RegisterButton`
- 必须挂 `Button`
- 文案固定为：`注册并登录`
- 不允许依赖运行时补按钮逻辑

视觉规则：

- 白底蓝边
- 视觉弱于主按钮
- 与主按钮同宽优先

### 8.5 StatusLabel

要求：

- 节点名必须为 `StatusLabel`
- 必须是纯展示节点
- 推荐挂 `Label`
- 不得是 `EditBox`
- 必须位于按钮组下方，属于 `LoginPanel` 内部

文案规则：

- 默认：`请输入账号和密码`
- 登录中：`正在登录，请稍候…`
- 成功：`登录成功，正在进入…`
- 账号错误：`账号或密码不正确`
- 网络错误：`网络有点慢，请稍后重试`

颜色规则：

- 默认灰
- 成功绿
- 错误红

### 8.6 LoadingNode

要求：

- 节点名必须为 `LoadingNode`
- 必须是纯展示节点
- 默认隐藏
- 必须属于 `LoginPanel` 内部

建议结构：

```text
LoadingNode
 ├── Dot1
 ├── Dot2
 └── Dot3
```

规则：

- 三圆点横向排列
- 循环跳动
- 只在提交过程中显示
- 不得做成输入框
- 不得与 `StatusLabel` 重叠

## 九、背景火狐狸落地规则

这是最容易跑偏的地方，所以单独写死。

### 9.1 性质定义

火狐狸属于：

**弱化角色剪影**

不是完整插画，不是叙事主视觉，不是世界观展示图。

### 9.2 必须满足

- 大体量
- 低透明
- 轻模糊
- 低细节
- 不遮挡功能区

### 9.3 不允许出现

- 清晰场景
- 道具
- 火焰特效
- 复杂毛发细节
- 高对比描边
- 强叙事动作

### 9.4 推荐构图

- 左侧偏中放置
- 大头优先，身体弱化
- 朝向登录区
- 与背景融在一起，不作为单独卡片或插画框出现

### 9.5 合格判断

合格状态：

- 用户先看到输入区和按钮
- 同时感知到左侧有一个大体量、柔和的火狐狸气质

不合格状态：

- 用户第一眼先在看狐狸细节
- 狐狸轮廓压住输入框
- 狐狸比按钮更抢

## 十、Prefab 建议

当前阶段不必过度拆分，但建议至少做这 3 类可复用结构：

1. `InputFieldBase`
2. `PrimaryButton`
3. `SecondaryButton`

`StatusLabel` 和 `LoadingNode` 当前阶段可以先留在场景里，不必急着 prefab 化。  
这是因为它们和登录流绑定更紧，先保证固定结构更重要。

## 十一、脚本绑定蓝图

推荐 `LoginController` 暴露如下引用：

- `usernameInput: EditBox`
- `passwordInput: EditBox`
- `loginButton: Button`
- `registerButton: Button`
- `statusLabel: Label`
- `loadingNode: Node`

控制职责只做这些：

- 读取输入
- 校验空值
- 切换按钮可用状态
- 设置状态文案
- 控制 `LoadingNode` 显隐
- 调用登录 / 注册接口
- 登录成功后切场景

不负责：

- 动态创建按钮
- 动态创建输入框
- 动态拼整页布局

## 十二、状态切换规则

### 默认态
- `StatusLabel = 请输入账号和密码`
- `LoadingNode = 隐藏`

### 提交中
- `StatusLabel = 正在登录，请稍候…`
- `LoadingNode = 显示`
- 按钮可临时禁用

### 成功态
- `StatusLabel = 登录成功，正在进入…`
- `LoadingNode = 显示或短暂显示`
- 执行跳转

### 失败态
- `LoadingNode = 隐藏`
- `StatusLabel = 对应错误原因`

## 十三、动画落地范围

只允许这 3 个：

1. `LoginButton` 按压回弹
2. `Logo` 轻呼吸
3. `LoadingNode` 三点循环

禁止新增：

- 背景粒子
- 火狐狸摆动
- 页面整体浮动
- 大量入场动画

## 十四、冒烟测试蓝图

执行完成后，至少要过这 8 条：

1. 场景中存在 6 个关键节点：
   - `UsernameInput`
   - `PasswordInput`
   - `LoginButton`
   - `RegisterButton`
   - `StatusLabel`
   - `LoadingNode`
2. `StatusLabel` 在 `LoginPanel` 内部，而不是页面底部独立节点
3. `LoadingNode` 在 `LoginPanel` 内部，默认隐藏
4. 输入账号密码过程稳定，不出现焦点丢失、输入中断
5. 登录按钮可点击，并触发状态切换
6. 注册按钮为固定场景节点，不依赖运行时补建
7. 火狐狸不遮挡输入区和按钮区
8. 登录区第一视觉仍然清晰可用，而不是被背景吞没

## 十五、登录页美术制作指导书 V1.0

### 1. 制作目标

本页不是“展示插画”，而是“可用界面”。

视觉目标：  
**轻玩具感 + 克制的软 UI + 高可用性**

### 2. 风格总则

#### 2.1 关键词
- 软：大圆角、无尖锐边
- 稳：不跳、不炸、不花
- 干净：少装饰、少纹理
- 有触感：轻厚度（微阴影 / 高光）

#### 2.2 明确禁止
- 不做“派对游戏菜单风”
- 不做高饱和糖果配色堆叠
- 不做复杂场景插画
- 不做高细节写实 / 毛发

#### 2.3 色彩体系
- 主色（UI）：蓝青系（用于主按钮 / 高亮）
- 中性：白 / 蓝灰（输入框 / 文字）
- 点缀（角色）：柔橙（火狐狸）

原则：  
**UI 冷色 + 角色暖色 = 平衡**

### 3. 组件清单与产出物

本页美术只需要产出以下资源：

必做：
- 输入框底板（通用）
- 主按钮（登录）
- 次按钮（注册）
- Loading 三圆点（或单点 + 动效序列）
- Logo 区（标题 + 小宠物头）
- 火狐狸背景角色（弱化剪影版）

不做：
- 主界面卡片
- 复杂插画
- 粒子素材
- 动态骨骼动画

### 4. 输入框制作规范

#### 4.1 外形
- 形状：圆角矩形
- 圆角：约等于高度的一半
- 边框：清晰但低对比（蓝灰）

#### 4.2 视觉表现
- 背景：纯白或极浅灰
- 阴影：轻（仅一层）
- 不使用高光

#### 4.3 触感目标
看起来像“柔软塑料块”，但仍然是功能控件。

#### 4.4 状态要求
需兼容两种视觉：
- 默认态（浅边框）
- Focus 态（边框变蓝 + 轻微发光）

#### 4.5 禁止
- 不做玻璃质感
- 不做重渐变
- 不做内阴影复杂结构

### 5. 按钮制作规范

#### 5.1 主按钮（LoginButton）

目标：
页面唯一视觉重心（除角色外）

外观：
- 圆角明显比输入框更圆
- 渐变：蓝 -> 青
- 顶部轻微高光带
- 底部轻微阴影

触感目标：
看起来“可以按下去”，但不过度果冻化。

按压状态：
- 轻微压扁
- 亮度略降低

#### 5.2 次按钮（RegisterButton）

外观：
- 白底
- 蓝色描边
- 无高光
- 阴影极弱或无

原则：
明显弱于主按钮，不争视觉焦点。

### 6. Loading 制作规范

#### 6.1 表现形式
三个圆点（等间距横排）

#### 6.2 视觉
- 圆点颜色：灰或主色蓝
- 不做渐变
- 不做发光

#### 6.3 动效交付
可选两种方式：
- 提供静态点（程序做动画）
- 提供序列帧（点逐个跳）

#### 6.4 禁止
- 不做爪印
- 不做复杂 loading 动画

### 7. Logo区制作规范

#### 7.1 结构
`[圆脑袋小宠物] + 学伴精灵（文字）`

#### 7.2 文字
- 字体圆润
- 可加轻描边或轻阴影（非常弱）

#### 7.3 小宠物头
- 形状：圆头
- 表情：简单（微笑）
- 细节：极简

#### 7.4 动效适配
需要适合“呼吸缩放”。

### 8. 火狐狸制作规范（重点）

#### 8.1 角色性质
火狐狸是“背景弱化剪影”，不是主视觉插画。

#### 8.2 形态要求
- 大头优先
- 身体弱化
- 轮廓清晰
- 边缘圆润

#### 8.3 表情
- 微笑 / 平静
- 不夸张
- 不张嘴大表情

#### 8.4 构图
- 左侧偏中
- 可裁切
- 朝向登录区

#### 8.5 视觉处理
- 透明度：`20% ~ 35%`
- 模糊：轻
- 饱和度：略降

#### 8.6 必须避免
- 场景（地面 / 环境）
- 道具（书本 / 火焰特效）
- 复杂毛发
- 强高光
- 描边轮廓

#### 8.7 合格判断

合格：
- 远看能认出是狐狸
- 近看不抢 UI

不合格：
- 细节丰富到可以“欣赏”
- 比按钮更吸引注意力

### 9. 背景制作规范

#### 9.1 基础
- 渐变背景（浅蓝 -> 更浅蓝）
- 不使用纹理贴图

#### 9.2 装饰
可加：
- 2~3 个模糊光斑
- 极弱圆形块

#### 9.3 禁止
- 插画背景
- 云朵细节
- 场景透视
- 图案堆叠

### 10. 尺寸与留白原则

#### 10.1 组件比例
- 输入框：高、偏胖
- 按钮：更高、更胖
- 圆角：统一偏大

#### 10.2 间距节奏
- 模块间距：大
- 输入框间距：中
- 按钮间距：中偏小

#### 10.3 总体感觉
“松、稳、呼吸感强”

### 11. 交付规范

#### 11.1 文件格式
- PSD / Figma（推荐 Figma 组件化）
- 分层清晰

#### 11.2 命名规范
- `Input_Base`
- `Button_Primary`
- `Button_Secondary`
- `Loading_Dot`
- `Fox_Background`
- `Logo_Mascot`

#### 11.3 切图规范
- 导出 PNG（透明背景）
- 分辨率：1x / 2x（按项目需求）
- 不合并不同功能层

#### 11.4 不允许
- 扁平化整张图
- 把多个组件做成一张不可拆分图片

### 12. 最终验收标准

必须满足：

1. 功能优先
   - 输入框清晰
   - 按钮明确
   - 状态易读

2. 风格统一
   - 所有组件圆角一致
   - 色彩体系一致
   - 阴影一致

3. 角色不过界
   - 火狐狸有存在感
   - 但不干扰操作

4. 无“廉价感”
   - 不花
   - 不乱
   - 不重

## 十六、与当前代码现态的对齐提醒

当前 [LoginController.ts](E:/buddy-client/assets/scripts/ui/login/LoginController.ts) 仍保留了两类运行时兜底：

1. 场景中找不到 `RegisterButton` 时，会运行时补建一个注册按钮
2. 会通过节点名查找 `UsernameInput`、`PasswordInput`、`StatusLabel`、`LoadingNode`

所以执行时要特别注意：

- **目标不是只把节点摆进场景里**
- 而是要让场景中的这套节点，真正成为脚本实际使用的唯一节点

当前推荐收口标准：

1. 场景中已存在 `RegisterButton`
2. 运行时不再触发“补建注册按钮”
3. `StatusLabel` / `LoadingNode` 都由场景固定节点承载
4. 登录、注册、状态切换不低于当前基线

**先把登录页做成“场景里本来就有、脚本只负责用”的状态，不要再让脚本补整套 UI。**

## 十八、当前已知待优化项

以下问题已确认存在，但当前阶段不作为阻塞项，先记录，后续统一调整：

1. 背景渐变仍有明显色阶感
   - 现态渐变不够平滑，能看到分层或像素拼接感
